# 10 Artificial Intelligence for Robotics

## Why this matters
Classical robotics gives you accurate perception and controllable motion, but it doesn't tell the robot *what to do* when the state space is too large or too messy to hand-code with rules. Learning-based decision-making — reinforcement learning, imitation learning, and now large pretrained "generalist" policies — is how robots pick up behaviors that are hard to specify by hand: dexterous manipulation, locomotion over rough terrain, or following an open-ended natural-language instruction. This file sits above 07perception.md: it assumes you already have images, point clouds, or detections coming in, and focuses on the layer that turns that information into decisions and actions. It's also the layer with the biggest gap between "works in simulation" and "works on hardware," so a good chunk of the field is really about closing that gap.

## Core concepts
- **Markov Decision Process (MDP)** — the formal framework (states, actions, transitions, reward) underneath almost all RL; understanding it makes every algorithm below easier to reason about.
- **Reinforcement learning (RL) basics** — an agent learns a policy by trial and error, maximizing cumulative reward; know the difference between value-based (Q-learning, DQN), policy-gradient (REINFORCE, PPO), and actor-critic (SAC, TD3) families and when each is a reasonable default.
- **Reward design and shaping** — in robotics the reward function is usually the hardest part to get right; sparse rewards, reward hacking, and shaping pitfalls are worth understanding before you write your first one.
- **Imitation learning / behavioral cloning** — training a policy directly on expert demonstrations (state→action pairs) via supervised learning; simpler and more sample-efficient than RL but suffers from compounding error and distribution shift (see DAgger as a fix).
- **Learning from demonstration (LfD) and teleoperation** — how demonstrations actually get collected on real robots (kinesthetic teaching, VR/leader-follower teleop, motion capture) — data collection is often the real bottleneck, not the algorithm.
- **Sim-to-real transfer** — training in simulation is cheap and safe, but policies overfit to simulator quirks; domain randomization (randomizing textures, physics, sensor noise) and system identification are the standard mitigations.
- **Simulation environments for learning** — Gymnasium (successor to OpenAI Gym) as the standard RL environment API; robot-specific simulators such as NVIDIA Isaac Lab / Isaac Sim, MuJoCo, and Gazebo/Gazebo Classic for physics-accurate training at scale, often with GPU-parallelized environments for faster training.
- **Foundation models and Vision-Language-Action (VLA) models** — large models pretrained on internet-scale (and increasingly robot-scale) data that map camera images plus a language instruction directly to robot actions, aiming for behaviors that generalize across tasks and embodiments rather than one policy per task.
- **Robot learning datasets** — cross-embodiment datasets (e.g. Open X-Embodiment) that aggregate demonstrations from many robots/labs specifically to pretrain generalist policies, analogous to how vision/language models pretrain on large web-scale corpora.
- **Behavior trees and finite state machines as an alternative to learning** — not every decision layer needs to be learned; behavior trees (see 04intermediateros2.md) are often the more debuggable, production-ready choice for structured tasks, and hybrid architectures (learned low-level skills, scripted high-level logic) are common in practice.
- **Model-based RL and world models** — learning a predictive model of environment dynamics and planning against it, which can be far more sample-efficient than model-free RL — important because real robots can't afford millions of trial-and-error episodes.
- **Offline / batch RL** — learning a policy purely from a fixed dataset of past interactions, without further environment access, which matters when live data collection on hardware is expensive or risky.
- **Evaluation and safety** — how to know a learned policy is actually good before it touches hardware: held-out task success rate, sim benchmarks, and safety constraints/action limits as guardrails around a policy you don't fully trust.
- **Edge and on-device inference** — running a trained model (not training it) on constrained onboard compute (e.g. Jetson-class boards, Raspberry Pi with a TPU accelerator); techniques like quantization and pruning trade a little accuracy for a lot of speed and memory headroom.

## Resources
- Sutton & Barto, *Reinforcement Learning: An Introduction* — the standard free textbook for RL fundamentals (search "Sutton Barto reinforcement learning book," a free PDF is hosted by the authors).
- Gymnasium documentation (gymnasium.farama.org) — the maintained fork of OpenAI Gym; the reference API for RL environments and a good place to prototype algorithms before touching a real robot.
- NVIDIA Isaac Lab documentation (search "NVIDIA Isaac Lab docs") and Isaac Sim documentation — GPU-accelerated robot simulation and RL training built on NVIDIA Omniverse, the current mainstream option for large-scale sim training.
- MuJoCo documentation (mujoco.readthedocs.io) — the physics engine widely used in manipulation and locomotion RL research, now open-source under DeepMind.
- Stable-Baselines3 documentation (stable-baselines3.readthedocs.io) — a well-maintained, well-documented set of reference RL algorithm implementations (PPO, SAC, TD3, DQN) that's a reasonable starting point instead of implementing from scratch.
- Hugging Face LeRobot (search "Hugging Face LeRobot") — an open-source library aimed specifically at real-world robot learning: imitation learning, datasets, and pretrained policies with a low barrier to entry.
- Open X-Embodiment project and paper (search "Open X-Embodiment robotics dataset") — the large cross-embodiment dataset and associated RT-X models that catalyzed a lot of current VLA work.
- Key papers worth reading by title: "Playing Atari with Deep Reinforcement Learning" (DQN, Mnih et al.), "Proximal Policy Optimization Algorithms" (Schulman et al., PPO), "A Reduction of Imitation Learning and Structured Prediction to No-Regret Online Learning" (Ross et al., DAgger), "RT-2: Vision-Language-Action Models Transfer Web Knowledge to Robotic Control" (Google DeepMind), "Domain Randomization for Transferring Deep Neural Networks from Simulation to the Real World" (Tobin et al.) — search arxiv.org for each by title.
- ROS 2 integration: search "ros2_control" and your simulator's ROS 2 bridge docs (e.g. Isaac Sim ROS 2 bridge, Gazebo ros_gz) for wiring a learned policy's action output into an actual robot control loop.
- TensorFlow Lite / LiteRT and PyTorch Mobile documentation — for deploying a trained model to run inference on constrained onboard hardware.

## Hands-on checkpoints
- [ ] Solve a classic control task (CartPole, then something continuous like Pendulum or LunarLanderContinuous) in Gymnasium with a Stable-Baselines3 algorithm to get the RL training loop under your fingers.
- [ ] Write your own reward function for a simple custom Gymnasium environment and observe how a bad reward shape produces unintended ("reward hacking") behavior.
- [ ] Collect a small teleoperated demonstration dataset for a simple task in simulation and train a behavioral-cloning policy on it; compare its failure modes to the RL policy above.
- [ ] Reproduce compounding error from behavioral cloning on purpose (start the policy from an off-distribution state and watch it fail), then read about DAgger as the fix.
- [ ] Train a policy in a robot-specific simulator (Isaac Lab, MuJoCo, or Gazebo) on a locomotion or simple manipulation task, using domain randomization on at least one physical parameter.
- [ ] Deploy a trained policy's inference step onto the actual perception/control pipeline from 07perception.md and 04intermediateros2.md, closing the loop from sensor to learned policy to actuator command.
- [ ] Try an existing pretrained VLA or generalist policy (e.g. via LeRobot) on a simulated or real manipulation task and evaluate how well it generalizes to a variant it wasn't explicitly trained on.
- [ ] Quantize or otherwise optimize a trained model and benchmark inference latency/memory on constrained hardware versus a full-size desktop/GPU run.
