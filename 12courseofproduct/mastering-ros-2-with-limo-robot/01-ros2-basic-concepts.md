# Mastering ROS 2 with LIMO-Robot — Unit 1: ROS2 Basic Concepts

Every later unit in this course — mapping, localization, navigation — is built out of ROS2 packages, nodes, and launch files. This unit establishes that vocabulary and the workspace habits you'll use for the rest of the course, using the LIMO's own software stack as the running example.

## Workspaces and packages

A ROS2 workspace is just a directory with a `src/` folder containing packages, plus `build/`, `install/`, and `log/` directories generated by the build tool. Nothing magic — `colcon` inspects `src/` for packages and builds them into `install/`. A package is the unit of distribution: it has a `package.xml` (name, dependencies, maintainer) and either a `CMakeLists.txt` (C++/ament_cmake) or a `setup.py` (Python/ament_python).

```bash
mkdir -p ~/limo_ws/src
cd ~/limo_ws/src
ros2 pkg create --build-type ament_python limo_lessons --dependencies rclpy
```

This scaffolds `limo_lessons/` with a `package.xml`, `setup.py`, and a `limo_lessons/` module directory where your node code lives. Keep one package per logical capability (e.g. `limo_navigation`, `limo_perception`) rather than one giant package — it makes dependencies and launch files easier to reason about later.

## Nodes and the ROS2 graph

A node is a single-purpose process that does one job — reading a sensor, running a controller, publishing a transform. Nodes discover each other automatically at runtime via DDS (the middleware ROS2 sits on top of); there is no central master process like there was in ROS1. The set of running nodes and how they're connected (topics, services, actions) is called the **ROS2 graph**, and you can inspect it live:

```bash
ros2 node list
ros2 node info /limo_base_node
```

A minimal Python node looks like this — note it does nothing on its own until you `spin()` it, which hands control to the executor that processes callbacks:

```python
import rclpy
from rclpy.node import Node

class HelloLimo(Node):
    def __init__(self):
        super().__init__('hello_limo')
        self.get_logger().info('LIMO node is up')

def main():
    rclpy.init()
    node = HelloLimo()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()
```

## Compilation with Colcon

`colcon` is the successor to `catkin_make`. It walks `src/`, figures out the dependency order between packages, and builds each with the right backend (CMake or Python setuptools):

```bash
cd ~/limo_ws
colcon build --symlink-install
source install/setup.bash
```

`--symlink-install` is worth using by default during development: for Python packages it symlinks your source files into `install/` instead of copying them, so edits to a Python node take effect without rebuilding. C++ nodes still need a rebuild after source changes. `colcon build --packages-select limo_lessons` rebuilds just one package when your workspace grows large.

## Launch files

A launch file starts multiple nodes together with their parameters, remappings, and conditional logic — you rarely start LIMO's stack node-by-node by hand. ROS2 launch files are themselves Python, which means you get real conditionals and loops instead of a static XML tree:

```python
from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
    return LaunchDescription([
        Node(
            package='limo_lessons',
            executable='hello_limo',
            name='hello_limo',
            output='screen',
        ),
    ])
```

```bash
ros2 launch limo_lessons hello_limo.launch.py
```

## Try it yourself

Create a package called `limo_lessons` as shown above, add the `hello_limo` node to `setup.py`'s `console_scripts` entry points, build it with colcon, and write a launch file that starts it with a remapped node name (e.g. `name='hello_limo_2'`). Confirm with `ros2 node list` that your remapped node appears under the new name.
